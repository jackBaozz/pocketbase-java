# pocketbase-java 技术架构与开发规范

## 1. 项目简介

`pocketbase-java` 同时提供 PocketBase Java SDK 和一个 PocketBase 风格 embedded server。项目目标是为 Java 应用提供稳定、低依赖、可测试、易扩展的 PocketBase 客户端基础能力，并提供可 GraalVM Native Image 打包的本地服务端实现。

本项目不引入 Spring Runtime，也不绑定复杂 Web 容器。服务端使用 JDK `HttpServer`，Admin UI 使用 `UI/` 下的 React/Vite 工程构建为静态资源，数据以 JSON 文件持久化，优先保证 native-image 可控性和依赖体积。

## 3. 架构特性

### 3.1 SDK 分层

项目采用轻量分层，不引入复杂框架：

- **Client 层** (`PocketBaseClient`): 维护基础 URL、HTTP 客户端、JSON 序列化、认证状态和通用请求发送逻辑。
- **Service 层** (`CollectionService`, `CollectionsService`): 按 PocketBase 资源域封装 API。
- **Model 层** (`AuthResponse`, `RecordList`, `PocketBaseError`): 描述 PocketBase 常用响应结构。
- **Options 层** (`ListOptions`, `RecordQuery`): 描述请求查询参数，避免调用方手写散乱 Map。
- **Exception 层** (`PocketBaseException`): 统一包装 HTTP 错误和传输错误。

### 3.2 低依赖设计

SDK 不直接依赖 Spring、OkHttp、Retrofit 等框架，避免把使用方锁死在特定生态内。HTTP 由 JDK 原生能力完成，运行时只引入 Jackson。

### 3.3 Embedded Server 分层

- **启动层** (`PocketBaseServer`, `LocalPocketBase`, `ServerConfig`): 解析 CLI 参数、启动 `HttpServer`、暴露可测试的程序化入口。
- **HTTP 层** (`HttpApi`): 处理 PocketBase 风格路由、JSON 响应、错误结构、CORS 和 Admin UI 静态资源。
- **存储层** (`JsonFileStore`): 管理集合 schema、记录数据、file storage、backup/restore、superuser bootstrap、auth 登录、access rules、简单查询和持久化。
- **上传层** (`MultipartFormData`, `UploadedFile`, `ThumbnailGenerator`): 解析 record create/update 的 `multipart/form-data`，将 file 字段落盘，并为配置过的图片生成按需缩略图。
- **Realtime 层** (`RealtimeHub`): 维护 SSE client、订阅 topic、官方 `subscriptions[]`/`options.query` 参数解析和记录变更事件分发。
- **安全层** (`PasswordHasher`, `TokenService`): PBKDF2 密码哈希和 HMAC-SHA256 token 签发/校验。
- **规则层** (`RuleEvaluator`, `RequestPrincipal`): 执行 collection access rules、API filter 的常用表达式子集和 `@collection.*` 聚合字段匹配。
- **模型层** (`CollectionSchema`, `FieldSchema`): 对齐 PocketBase collection/field 的核心字段。

### 3.4 GraalVM / Native Image 友好约束

虽然本项目不默认执行 Native Image 编译，但核心设计遵循 GraalVM 友好原则：

| 特性 | 约束说明 |
| --- | --- |
| 反射 | 不主动使用动态反射；JSON 映射集中由 Jackson 完成。 |
| 动态代理 | 不使用 JDK 动态代理或 CGLIB。 |
| 资源加载 | Admin UI 静态资源通过 `resource-config.json` 显式声明。 |
| 第三方依赖 | 新增运行时依赖前必须评估 Native Image 兼容性和依赖体积。 |
| HTTP 服务 | 使用 JDK HTTP server，避免 Servlet 容器和动态扫描。 |

## 6. 开发环境要求

### 6.1 JDK

- 最低版本: JDK 17。
- 推荐版本: JDK 21 或 Oracle GraalVM 25。
- `.java-version` 固定为 `17`，保证 SDK 面向使用方的最低兼容性。

### 6.2 Maven

- Maven 3.9+。
- 项目不包含 Maven Wrapper，默认使用本机全局 Maven。
- 如果中央仓库访问不稳定，使用项目内 `settings.xml`:

```bash
mvn -gs settings.xml -s settings.xml test
```

### 6.3 Admin UI

- UI 源码放在 `UI/`，使用 React + Vite + TypeScript。
- `UI/vite.config.ts` 设置 `base: "/_/"`，并将 `build.outDir` 指向 `../src/main/resources/pocketbase-admin`。
- 修改 UI 后运行：

```bash
cd UI
npm install
npm run build
```

默认 Maven 构建不自动执行 npm，避免 Java 测试和 native-image 构建被 Node 环境耦合；发布或打 native 前需要先提交最新 UI build output。

## 7. 构建、测试与验证

### 7.1 运行测试

```bash
mvn -gs settings.xml -s settings.xml test
```

### 7.2 打包

```bash
mvn -gs settings.xml -s settings.xml clean package
```

打包产物包括：

- `target/pocketbase-java-0.1.0-SNAPSHOT.jar`
- `target/pocketbase-java-0.1.0-SNAPSHOT-all.jar`
- `target/pocketbase-java-0.1.0-SNAPSHOT-sources.jar`
- `target/pocketbase-java-0.1.0-SNAPSHOT-javadoc.jar`

运行服务端：

```bash
java -jar target/pocketbase-java-0.1.0-SNAPSHOT-all.jar serve --http 127.0.0.1:8090 --dir pb_data
```

### 7.3 Native Image

```bash
mvn -gs settings.xml -s settings.xml -Pnative -DskipTests package
./target/pocketbase-java serve --http 127.0.0.1:8090 --dir pb_data
```

### 7.4 本地安装

```bash
mvn -gs settings.xml -s settings.xml clean install
```

## 8. 编码规范

### 8.1 API 设计规范

- 公共 API 必须保持小而稳定，新增能力优先通过新增方法，不随意改变已有方法语义。
- 请求参数使用 Options 对象封装，避免公共方法暴露散乱字符串 Map。
- 任意远程 API 错误必须保留原始响应体，方便排查 PocketBase 后端校验失败。
- SDK 不吞异常；非 2xx 响应统一抛 `PocketBaseException`。

### 8.2 命名规范

- 包名: `io.github.jackbaozz.pocketbase`。
- Client: 以 `Client` 结尾。
- 资源服务: 以 `Service` 结尾。
- 查询参数: 以 `Options` 或 `Query` 结尾。
- 远端响应结构: 使用名词，如 `RecordList`、`AuthResponse`。

### 8.3 依赖规范

- 新增运行时依赖前必须回答三个问题：
  1. 是否能用 JDK 标准库完成？
  2. 是否会显著增加传递依赖？
  3. 是否影响 GraalVM Native Image 兼容性？
- 测试依赖可以适度引入，但当前优先使用 JDK 内置 `HttpServer` 做 HTTP 行为测试。

## 9. 测试要求

新增或修改 SDK 行为时，至少覆盖：

- URL path 和 query 参数构造。
- JSON 请求体序列化。
- 认证 token 保存和自动注入。
- 非 2xx 响应异常映射。
- 空参数、非法参数的本地校验。

测试应验证真实 HTTP 请求行为，而不是只测纯对象拼装。

新增或修改 embedded server 行为时，至少覆盖：

- 真实 `HttpServer` 启动和 HTTP 请求。
- superuser bootstrap 和登录。
- auth collection 登录和 auth refresh。
- auth requestOtp/authWithOtp、verification、password reset、email change、impersonate 和 tokenKey 轮换失效。
- SDK 调用 embedded server 的兼容性。
- 集合创建、collection meta scaffolds/OAuth2 provider metadata、collections import/truncate、记录 CRUD、持久化重启读取。
- JSON/multipart Batch API record create/update/upsert/delete，以及失败时 records 和 storage 文件回滚。
- SQL API superuser-only、query 校验、SELECT 输出结构、collection 记录写入和多语句写事务回滚。
- relation `expand` 和目标记录 `viewRule` 过滤。
- multipart 文件上传、MIME/大小校验、文件名持久化、protected file token、`thumb` 缩略图和 `/api/files` 读取。
- backup 创建、列表、下载、还原和删除。
- settings 读取/更新、官方字段名兼容、敏感字段省略、S3/email/Apple client secret 测试接口和活动 logs 列表/单条/stats。
- crons 列表、缺失 job 404、`__pbAutoBackup__` 注册和手动触发自动备份。
- Realtime SSE 连接、JSON/query/form/multipart 订阅提交、`options.query` filter/expand/fields 和 create/update/delete 事件推送。
- Admin UI 静态资源可访问。
